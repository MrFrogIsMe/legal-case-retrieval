# 實驗 11：寬鬆相關性評估 — 翻案結論

> 對應腳本：experiments/11_relaxed_eval.py
> 前置：07（嚴格 single-relevant 消融）、10（HyDE 改寫，嚴格尺規下變差）
> 相關性定義：候選與 ground-truth「同案由」或「共享主要實體法條」（排除刑訴/民訴）

---

## 1. 動機：尺規與任務本質不匹配

實驗 07/10 在「single-relevant（只有原本那篇判決算對）」尺規下得到反直覺結果：
- 最佳 Recall@5 僅 0.173（rerank）
- HyDE query 改寫**反而變差**（baseline 0.140 → A rewrite 0.080 → B hyde 0.053）

歸因：query 由某判決 facts_summary 口語化生成，殘留指向「那一篇」的語意指紋；
任何「標準化/通用化」手段（改寫成法律用語）會抹掉指紋、讓 query 平均靠近
一群相似判決，故在「找指定那篇」尺規下傷害命中率。

但「類案檢索」的真實目標是「找一批相似案件」，非「找指定那篇」。
故改用寬鬆相關性，使評估尺規匹配任務本質。

---

## 2. 寬鬆相關性結果（同案由 / 共享主要法條算相關）

| 策略 | Recall@5 | MRR@10 |
|------|----------|--------|
| baseline 口語 (dense) | 0.913 | 0.810 |
| A rewrite (dense) | 0.927 | 0.880 |
| B hyde (dense) | 0.900 | 0.831 |
| baseline (rerank) | 0.920 | 0.831 |
| **A rewrite (rerank)** | **0.933** | **0.870** |

---

## 3. 兩個關鍵結論

### 3.1 系統一直都能找到「同類案件」
所有方法 Recall@5 從嚴格尺規的 0.14 躍升至 **0.9+**。
證明低分純粹來自「single-relevant 找指定那篇」的尺規過嚴，
而非檢索系統缺陷（self-retrieval sim=1.0 亦佐證索引正常）。

### 3.2 HyDE 改寫在正確尺規下「翻案」為有效（核心發現）
| query 策略 | 嚴格尺規 R@5 | 寬鬆尺規 R@5 |
|-----------|-------------|-------------|
| baseline 口語 | 0.140 | 0.913 |
| A rewrite | 0.080（傷害） | 0.927（幫助） |

同一個 HyDE 改寫，在兩種尺規下效果**完全反轉**。這證明：
- 之前「改寫變差」不是改寫方法爛，是評估尺規與任務不匹配。
- 一旦尺規對齊「找類似案件」的真實目標，「把口語標準化成法律用語」的價值
  顯現——因為標準化正是為了召回同類案件。
- A rewrite + rerank 達最佳（R@5 0.933），改寫補語意鴻溝、rerank 補精排。

---

## 4. 對「口語↔法律術語鴻溝」問題的最終回答

使用者口語 query 接不上法律用詞，**最有效的解法是 query 端改寫（rewrite）
而非清洗 doc 或換 embedding**；但前提是評估與產品目標都定位為「類案檢索
（找相似案件）」，而非「精確找指定單篇」。

實驗鏈否證了兩個直覺方向、確立一個正解：
- ✗ 清洗 embed 文字：無效（+0.007，雜訊非主因）
- ✗ 在嚴格尺規追高 single-relevant：方向錯誤（尺規不匹配任務）
- ✓ 寬鬆尺規 + query 改寫 + rerank：R@5 0.933

---

## 5. 方法論備註

- 評估集 query 由判決 facts_summary 生成，本身偏向 single-relevant；
  寬鬆相關性（同案由/同法條）是貼近 LeCaRD、COLIEE 等法律 IR benchmark 的做法。
- 策略 C（受控生成）因案由預判用字串比對不可靠（小三案誤判為社維法）暫棄用，
  未來可改以 dense top-1 的案由作預判再受控生成。
