# 實驗 07：檢索消融評估結果

> 對應腳本：experiments/07_eval.py、08_clean_ablation.py
> 評估集：eval.jsonl（50 份判決 × 3 口語 query = 150 對，gemini-3.5-flash 生成）
> 索引：81,644 筆地院一審刑事（dense=facts_summary，BM25=摘要+案由+法條號）
> ground truth：query 由該判決 facts_summary 生成，命中「該篇」算正確（嚴格 single-relevant）

---

## 1. 主消融表（Recall@5 / MRR@10）

| 方法 | Recall@5 | MRR@10 | 說明 |
|------|----------|--------|------|
| Dense (BGE-M3) | 0.140 | 0.090 | 純語意向量 |
| BM25 (含法條號) | 0.000 | 0.000 | 純稀疏；口語 query 字面零重疊 |
| Hybrid (RRF) | 0.100 | 0.071 | dense+BM25 融合 |
| **Hybrid + Rerank** | **0.173** | **0.142** | **+ bge-reranker-v2-m3 精排，最佳** |

---

## 2. 關鍵發現

### 2.1 Rerank 顯著有效（核心結論）
- Recall@5：0.100 → 0.173（+73% 相對）
- MRR@10：0.071 → 0.142（翻倍）
- 原因：hybrid 召回的 top-20 候選中約 23% 含正確判決，reranker 用 (query, doc)
  交互打分把它們從後段拉到前段。MRR 翻倍佐證「命中項排名明顯前移」。
- 對齊 design_v1 第 3.2 節 D：reranking 的設計價值由數據證實。

### 2.2 純口語 query 下，BM25 反而有害
- BM25 Recall@5 = 0.000：口語 query（「小三」「求原諒」）與判決書正式用語字面零重疊，
  BM25 token 比對完全失效。
- Hybrid (0.100) < Dense (0.140)：混入 BM25 的雜訊排名稀釋了 dense 的好結果。
- 啟示：BM25 的價值在「使用者 query 帶法條號/明確關鍵字」時才顯現；
  純口語場景應以 dense 為主、rerank 精排。

### 2.3 清洗 embed 文字無顯著效果（否證假設，08_clean_ablation）
| dense embed | Recall@5 | MRR@10 |
|-------------|----------|--------|
| 原始 facts_summary | 0.140 | 0.090 |
| 清洗後（去日期/法條/金額） | 0.147 | 0.101 |
| Δ | +0.007 | +0.012 |
- 去除人名/日期/法條等「檢索雜訊」幾乎無提升 → 雜訊非低分主因。
- 教訓：中文人名 regex 清洗會誤傷常用字（「過程平和」→誤切），已棄用人名清洗。

---

## 3. 絕對值偏低的歸因（誠實討論）

最佳 Recall@5 僅 0.173，主因為任務本質難度，非系統缺陷：

1. **8 萬筆同質語料 + 嚴格 single-relevant**：同案由有數百至數千份高度相似判決，
   要命中「指定那一篇」極難。診斷顯示 dense top-20 僅命中 23%（34/150），
   即 77% 的正確判決連召回階段都進不了候選池——這是召回上限，rerank 無法補救。
2. **口語↔書面語語意鴻溝**：BGE-M3 對「小三」vs「與配偶發生不倫關係」的
   向量距離大於預期（design 第 1.2 節指出的核心難題）。
3. self-retrieval sanity check：用 facts_summary 原文檢索命中自己 sim=1.0，
   證明索引與檢索機制正常，低分來自 query↔doc 鴻溝而非 bug。

---

## 4. 下一步候選（提升召回，非精排）

rerank 已將「召回到的」排好；要再提升須從召回端著手：
- **Query 端術語正規化**（design 3.2 A，未實作）：口語→法律用語，拉近 query↔doc。
- **放寬相關性為「同案由/同法條」**：符合「類案檢索」本質（找相似案件而非唯一篇），
  作為輔助指標呈現。
- 兩者預期能把召回 23% 的天花板抬高，才是絕對值提升的關鍵。
