# 產品設計 v1：法律判決類案搜尋系統

> 對應設計文件：`docs/design_v1.md`（技術架構）
> 本文件聚焦「最終產品形態、功能、介面、使用者需求」

---

## 1. 最終呈現形態

完整工程（路線 B）的最終交付分兩條腿：

```
給「人」看的：可公開訪問的網站 demo
  使用者輸入事由 → 類似案例 + 法律分析 + 群體統計 + 誠實的免責揭露

給「學術/評分」看的：實驗報告 + 系統架構 + CI/CD pipeline
  消融實驗數據、可信度量化、技術文件
```

定位提醒：本課程為「生成式 AI 的人文導論」，**人文反思比重要足**。
不是「做了一個搜尋引擎」，而是「做了一個搜尋引擎，並誠實揭露它的邊界與偏差」——
這個揭露要做進介面，不是藏在報告附錄。

---

## 2. 功能清單

### 基礎功能（必做）

1. 事由輸入 → 類案檢索：口語事由回傳 top-5 相似判決
2. 案例卡片：案由、法院、事實摘要、判決結果、賠償（若有）
3. 法律分析：可能涉及法條、過失/故意、刑事 vs 民事
4. 免責與偏差揭露：主動說明「純財損刑事多不起訴」等事實

### 進階功能（加分）

5. Citation 高亮：標出結論依據原文哪一段，可疑引用標紅
6. 信心分數：每個抽取欄位顯示信心（高/中/低待確認）
7. 篩選與排序：依年份、法院、判決結果篩選
8. 統計視覺化：判決結果分布、賠償金額區間圖表

### 展示型功能（報告/demo 用）

9. 「系統如何運作」透明頁：攤開檢索過程
10. 實驗結果頁：消融表格網頁化

## 3. 介面設計：四方向疊成單頁漸進揭露

採 **A + B + C + D** 四個呈現方向，疊成由淺到深的單一介面。

### 四個方向

- **方向 A — 對話式情境引導**：不是冷搜尋框，而是像跟法律助理聊天，主動追問關鍵要件（「有人受傷嗎？」），自然帶出法律知識，呼應 agentic AI 主題。
- **方向 B — 案例對比視覺化**：使用者事由 vs 案例並排，標出「哪裡像、哪裡不像」，做出可解釋性。
- **方向 C — 判決結果儀表板**：群體統計圖表（「200 件中 60% 不受理…」），呈現分布而非單點預測，更誠實。
- **方向 D — 透明推理過程**：攤開 agent 思考鏈（理解→推法條→檢索→重排），把黑箱變透明。

### 分層結構

```
第 0 層  對話引導入口（A）        「跟我說說發生什麼事？」+ 追問要件
第 1 層  結果總覽（C）            白話摘要 + 判決分布圖 + 賠償區間
第 2 層  案例卡片列表             top-5，白話摘要，[展開]
第 3 層  案例詳情（B + citation） 對比表、原文高亮、可疑標紅、信心分數、完整法條
側/底    推理過程（D，預設折疊）   [1]理解→[2]推法條→[3]檢索→[4]重排
```

設計要點：
- 表層全白話（Persona 1 不被術語嚇到）
- 深層才出現法條、citation、信心分數（Persona 2 點開才看到）
- 方向 D 預設折疊，不打擾一般使用者

## 4. 使用者需求分群（Personas）

| Persona | 是誰 | 需求 | 對應功能 |
|---------|------|------|---------|
| 1 法律門外漢（主） | 你同學的情境 | 看得懂、有方向、知道嚴重程度 | 1-4、8 |
| 2 有法律基礎者 | 想查特定法條/看原文 | 檢索精準、能溯源、能篩選 | 5-7 |
| 3 評審/同學/老師 | 看技術深度與人文思考 | 系統怎麼運作、實驗數據 | 9-10 |

### 兼顧 Persona 1 + 2：漸進揭露（progressive disclosure）

不做「門外漢/專業模式」切換按鈕（逼使用者在不懂時先選擇）。
正確做法：**預設給門外漢看得懂的，專業細節藏在一層之下，想看再點開。**

- Persona 1 在表層（第 0-1 層）就滿足
- Persona 2 往下點一層（第 3 層）拿到精準與溯源
- 兩者共用同一條資料流，只是看的深度不同，不做兩套

### Persona 2 的輔助入口

進階使用者有時直接想用法條查（非從事由出發）：
主搜尋框旁放小連結「我知道法條/案由」→ 點開可輸入「刑法 284」或勾選案由篩選。
滿足精準需求，又不佔門外漢主視線。

## 5. 對應後端 API

漸進揭露讓後端不需為兩種人開兩套，而是同一份資料分層回傳：

```
POST /api/v1/clarify        對話追問（方向 A）— 表層
POST /api/v1/search         回傳 cases + stats + analysis（A 完成後一次給齊）
GET  /api/v1/case/{jid}     案例詳情 + citation 高亮（第 3 層，點開才載入）
POST /api/v1/search/trace   推理過程（方向 D，展開才打）
GET  /api/v1/stats          群體統計（方向 C 圖表）
GET  /api/v1/health         健康檢查
```

效能設計：
- `/search` 一次給「摘要+統計+案例列表」，前端控制揭露節奏（首屏快）
- `/case/{jid}` citation 高亮較重（回原文比對），lazy load
- `/search/trace` 同理，展開才打

---

## 6. 開發優先級

A+B+C+D 全做 + 兼顧兩 persona 工程量不小（A 對話、B 對比各需多一輪 LLM）。建議排序：

```
必做（MVP）：C 統計總覽 + 案例列表 + D 推理過程（便宜好做）
高價值：    A 對話引導（最有亮點）
有餘力：    B 對比表（最花工，解釋性最強）
```

先把 C+列表+D 做到能跑（完整可 demo），再加 A，行有餘力做 B。
即使時間不夠，也有完整可展示的東西。
